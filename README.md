# AWS EKS Platform

A production-grade AWS infrastructure platform built with Terraform, designed to deploy and manage a containerised workload on Elastic Kubernetes Service (EKS) across multiple environments.

## Architecture Overview

This platform provisions a complete AWS infrastructure stack covering networking, compute, storage, security, DNS, and observability — all built from reusable Terraform modules and deployed independently across dev, staging, and prod environments.
```
aws-eks-platform/
├── modules/
│   ├── vpc/          # Multi-AZ networking, NAT gateways, VPC flow logs
│   ├── eks/          # EKS cluster, node groups, OIDC, KMS encryption
│   ├── iam/          # Cluster roles, node group roles, IRSA
│   ├── ecr/          # Container registries with image scanning
│   ├── rds/          # PostgreSQL with encryption and Secrets Manager
│   ├── route53/      # Public and private hosted zones
│   └── monitoring/   # Prometheus, Grafana, CloudWatch alarms
└── environments/
├── dev/          # Single NAT, smaller nodes, no Multi-AZ RDS
├── staging/      # Multi-AZ, medium nodes, tighter alarm thresholds
└── prod/         # Full HA, larger nodes, 30-day metric retention
```

## Key Design Decisions

### Multi-environment with isolated state
Each environment (dev/staging/prod) has its own Terraform state file stored in S3 with DynamoDB locking. No shared state means no risk of one environment's changes affecting another.

### Single NAT gateway in dev, one per AZ in prod
Dev uses a single NAT gateway to reduce cost. Staging and prod use one NAT gateway per availability zone for resilience. Controlled by the `single_nat_gateway` variable.

### IRSA for pod-level IAM permissions
IAM Roles for Service Accounts (IRSA) allows individual Kubernetes pods to assume IAM roles directly rather than inheriting broad node-level permissions. This enforces least-privilege security at the pod level.

### KMS encryption everywhere
EKS secrets and RDS storage are encrypted with dedicated KMS keys with automatic key rotation enabled. Separate keys per service means a compromised key does not affect other services.

### Immutable image tags in prod
ECR repositories in prod use IMMUTABLE image tags — once pushed, a tag cannot be overwritten. This prevents accidental or malicious replacement of a known-good image.

### Lifecycle policies on ECR
Untagged images are deleted after 7 days. Only the last 10 tagged images are retained. This prevents storage costs accumulating from unused images over time.

### RDS password managed by Secrets Manager
The RDS master password is generated by Terraform, stored in Secrets Manager, and never exposed in state files or environment variables. Applications retrieve it at runtime via the AWS SDK.

## Module Structure

Each module follows a consistent four-file pattern:

| File | Purpose |
|------|---------|
| `versions.tf` | Terraform and provider version pinning |
| `variables.tf` | All inputs with descriptions and validation |
| `main.tf` | Resource definitions |
| `outputs.tf` | Values exported to calling environments |

## Environment Differences

| Setting | Dev | Staging | Prod |
|---------|-----|---------|------|
| NAT Gateways | 1 | 3 | 3 |
| Node type | t3.medium | t3.large | t3.xlarge |
| Node count | 2 | 3 | 5 |
| RDS Multi-AZ | No | Yes | Yes |
| RDS instance | db.t3.micro | db.t3.small | db.t3.medium |
| Metric retention | 7 days | 15 days | 30 days |
| CPU alarm threshold | 80% | 75% | 70% |
| Deletion protection | No | Yes | Yes |

## Getting Started

### Prerequisites
- Terraform >= 1.5.0
- AWS CLI configured with appropriate credentials
- An S3 bucket for remote state: `aws-eks-platform-terraform-state`
- A DynamoDB table for state locking: `aws-eks-platform-terraform-locks`

### Deploy dev environment

```bash
cd environments/dev
terraform init
terraform plan
terraform apply
```

### Deploy staging or prod

```bash
cd environments/staging   # or prod
terraform init
terraform plan
terraform apply
```

## Security Considerations

- EKS API server endpoint is private-only — no public access
- VPC flow logs enabled for all network traffic auditing
- IAM least-privilege enforced at cluster, node, and pod level
- All data encrypted at rest with KMS
- RDS accessible only from EKS node security group
- Secrets never stored in tfvars or state — managed via Secrets Manager

## Git Workflow

This repository follows a trunk-based development workflow with branch protection enforced on `main`.

- All changes are developed on feature branches (`feature/<description>`)
- A pull request is raised and must be reviewed and approved before merging
- Direct pushes to `main` are blocked by branch protection rules
- Commit messages follow the [Conventional Commits](https://www.conventionalcommits.org/) standard (`feat:`, `fix:`, `docs:`, `chore:`)

This mirrors the workflow used in professional engineering teams.
